Accessible Teaser Patterns Experiment

A teaser card with embedded media creates a conflict: mouse users want to click anywhere on the card to navigate; keyboard users need to reach the media controls independently. These three patterns explore the trade-offs.

Pattern A — Heading link only

The link lives only on the heading. "Read more" is a second link to the same destination — harmless visually, but creates a duplicate tab stop for keyboard and screen reader users.

Assistive technology behaviour

Mouse

Only the heading text and "Read more" are clickable. The image, body text, and the area around the video are dead zones.

Keyboard

Two tab stops for the same destination: the heading link, then the media controls, then "Read more". The duplicate link wastes a tab stop and adds confusion.

Screen reader

Announces both links separately — "Display Ergonomics at Small Sizes, link" and "Read more, link" — with no indication they go to the same place. Users must decide whether to follow both.

Verdict

Works without JavaScript. Acceptable for simple teasers without media, but the duplicate link is a meaningful friction point for AT users.

Pattern B — JS click delegation

A click listener on the wrapper navigates to the heading link's destination — unless the click landed on an interactive element (link, button, video controls). "Read more" is hidden from the accessibility tree to eliminate the duplicate.

Assistive technology behaviour

Mouse

Click anywhere on the teaser — image, text, space around the video — navigates. Clicking inside the video controls plays/pauses without triggering navigation.

Keyboard

One tab stop: the heading link. Media controls follow naturally. No duplicate "Read more" tab stop.

Screen reader

Announces one link: "Display Ergonomics at Small Sizes, link". "Read more" is removed from the accessibility tree via aria-hidden.

Requires JavaScript

Without JS the teaser degrades to Pattern A — the heading link still works, but the wrapper is no longer clickable.

Pattern C — Stretched link (CSS only)

A ::after pseudo-element on the heading link is stretched to cover the entire card via position: absolute; inset: 0. The video and any other interactive elements sit above it via z-index. No JavaScript required.

Assistive technology behaviour

Mouse

Click anywhere on the teaser navigates. Video controls sit above the overlay and work normally. No JavaScript required.

Keyboard

One tab stop: the heading link. Focus indicator appears on the heading text, not the card border — may feel disconnected from the card as a whole.

Screen reader

Announces one link: "Display Ergonomics at Small Sizes, link". Clean — no duplicates.

Caveat

Any other interactive elements inside the teaser (a second link, a button) need position: relative; z-index: 2 to sit above the overlay — easy to miss when adding content later. Also: if any ancestor gains a transform or opacity, the stacking context breaks and the z-index relationship may stop working.
