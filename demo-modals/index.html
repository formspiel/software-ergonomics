Modals & Pop-ups

An overview of patterns that surface content over or alongside a page — modals, pop-ups, popovers, tooltips, and alerts. Each pattern behaves differently with assistive technology. This page documents what screen readers announce, how keyboard navigation works, and when to use or avoid each pattern.

Modal Dialog Recommended

Assistive technology behaviour

What screen readers announce

When the dialog opens, screen readers announce the dialog role and its title (e.g. "Write a review, dialog"). The rest of the page is made inert — users cannot tab or read outside the dialog while it is open.

Keyboard behaviour

  • Tab / Shift+Tab — cycles through interactive elements inside the dialog only; focus cannot escape
  • Esc — closes the dialog and returns focus to the element that opened it

Nested dialogs

Click "Show guidelines" inside the dialog to open a second dialog on top. Esc closes only the topmost dialog; the one below remains open.

When to use

Use when a task requires the user's full attention or input before continuing. Avoid overusing modals — they interrupt workflow and can be disorienting for screen reader users if the purpose is unclear.

Implementation

Uses the native <dialog> element with showModal(). Focus trapping, Esc handling, ::backdrop, and page inert behaviour are all provided by the browser. Closing via a form submit button requires <form method="dialog"> — no JS needed for the close action itself.

Browser Popup Window Avoid

May be blocked by your browser's popup blocker.

Assistive technology behaviour

Known problems

  • Opens a new browser window — screen reader users may not notice the context switch
  • The relationship between the originating page and the new window is not communicated
  • Managing multiple windows is difficult for keyboard-only users
  • Can be blocked by popup blockers entirely, silently failing with no feedback
  • Visually resembles advertising popups — reduces trust

Recommendation

Avoid. Use a modal dialog or inline disclosure instead. If a new window is genuinely necessary (e.g. a PDF or an external tool), warn the user in advance with visible text such as "opens in a new window" and an icon.

Popover Recommended

This is a popover. It closes when you press Esc, click outside, or click the button again.

No JavaScript required — the browser handles everything via the popover and popovertarget attributes.

Assistive technology behaviour

What screen readers announce

When the popover opens, focus does not move automatically — the user must navigate to the popover content. Unlike a modal, the rest of the page remains accessible. Screen readers announce the content when the user tabs into it.

Keyboard behaviour

  • Esc — closes the popover
  • Clicking outside (light dismiss) — closes the popover
  • The trigger button toggles open/closed
  • Focus does not move automatically — the popover is non-modal

When to use

Use for supplementary information, contextual help, or action menus that do not require immediate interaction. The popover does not block the rest of the page, making it less disruptive than a modal dialog.

Implementation

Pure HTML — no JavaScript. Add popover to the target element and popovertarget="id" to the trigger button. Supported in all modern browsers (Chrome 114+, Firefox 125+, Safari 17+).

Abbreviation Tooltip Built-in

My ETA is 5:30 p.m.

Assistive technology behaviour

What screen readers announce

Most screen readers read the title attribute of <abbr> elements aloud — either alongside or in place of the abbreviation. The exact behaviour varies by screen reader and its settings.

Keyboard behaviour

The tooltip shown on mouse hover is not visible on keyboard focus in most browsers. Keyboard users and touch-screen users cannot access the tooltip.

Limitations

  • Not accessible on touch devices — no hover event
  • Not reliably visible for keyboard-only users
  • Screen reader behaviour is inconsistent across tools and settings

Recommendation

If the expansion is essential, write it inline on first use: Estimated Time of Arrival (ETA). Use <abbr> as a supplementary hint, not as the only source of the expansion.

Alert — ARIA Live Region Recommended

Assistive technology behaviour

What screen readers announce

Content injected into a role="alert" element is announced immediately by screen readers — without the user having to navigate there. The announcement interrupts whatever the screen reader is currently reading.

Keyboard behaviour

No keyboard interaction required. The alert appears in place and focus does not move. Sighted users see the message appear; screen reader users hear it spoken.

When to use

  • Validation errors after form submission
  • Status messages (save confirmed, operation failed)
  • Time-sensitive notifications

role="alert" vs role="status"

Use role="alert" (assertive) for urgent messages that interrupt. Use role="status" (polite) for non-urgent updates that wait for a pause in speech before announcing.

Implementation note

The element must be present in the DOM before content is injected. Inject text by setting textContent — not innerHTML. To re-announce the same message, clear the element first, then set it in the next animation frame.

Native Browser Alert Avoid

Assistive technology behaviour

What screen readers announce

Focus moves to the alert dialog and the message is read aloud. The browser's own dialog role is correctly communicated. Dismissal returns focus to the page.

Keyboard behaviour

Enter or Space dismisses the alert. The dialog appearance and text are controlled entirely by the browser and operating system — they cannot be styled or translated by the application.

Critical limitation

window.alert() blocks JavaScript execution on the page until dismissed. This can freeze animations, timers, and pending network requests.

Recommendation

Avoid in production. Use an ARIA live region for notifications or a native <dialog> for confirmations. Reserve window.alert() for quick debugging only.
