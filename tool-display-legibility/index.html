Cap-height ratios are approximations — published per-em values, not values measured at specific point sizes on specific platforms or rendering environments. Numbers will be in the right ballpark, but anyone doing certification work should verify by measuring on the actual target. The hinting quality badge indicates where extra caution is warranted.

px ↔ pt conversion uses the CSS convention 1 pt = 1.333 CSS px (96 ÷ 72). Native desktop applications that work with physical points may differ slightly from this figure.

Size accepts half steps (e.g. 10.5) because pt values often convert to non-integer CSS px — 8 pt = 10.667 CSS px — and CSS itself permits fractional font sizes. Whole numbers cover most real-world usage.

Measure & verify

Cap-height ratios are estimates. For ground truth: take a screenshot at 100 % browser zoom, measure the height of the capital E (the first character) in an image editor, then enter the result. The render target always uses a white background so it is easy to isolate in any screenshot tool.

Notes

About this calculator

The calculator answers the question: given a display and a viewing distance, how tall must a character be on screen to remain legible? ISO 9241-303 defines three arc-minute thresholds (16′ / 20′ / 22′) that translate viewing distance into a minimum physical character height — the calculator converts those into dpx and CSS px for your specific screen.

dpx means display pixels — physical pixels on the panel. It is distinct from CSS px, which the browser scales by the device pixel ratio (DPR). High-DPR screens (Retina, modern phones) render 1 CSS px as several dpx.

Cap-height ratio

The cap height of a font is the distance from the baseline to the top of a flat capital letter (E, H, I). It is the typographic measure that determines whether text meets the ISO 9241-303 minimum — the standard specifies arc-minute thresholds that convert to a minimum physical height in mm, which must be met by the cap height of the rendered character, not the full em size.

The cap-height ratio is cap height divided by the font size, both in the same unit:

cap height (CSS px) = font size (CSS px) × cap ratio
e.g. Arial 14 px × 0.716 = 10.0 CSS px cap height

Ratios typically fall between 0.63 and 0.73 depending on the typeface. A font with a high cap ratio (e.g. Verdana 0.728) produces taller capitals at the same point size than one with a low ratio (e.g. Calibri 0.638). This means two fonts set at the same CSS px size can differ by over 10 % in actual cap height — and therefore in ISO compliance.

The values stored in this calculator are published per-em approximations from font metrics, not values measured on a specific platform or rendering engine. Hinting, ClearType, anti-aliasing, and OS-level font scaling all affect the final rendered height. Use the Measure & verify section to determine the actual cap height on your screen.

The ratio concept applies to Latin, Cyrillic, and Greek fonts. CJK, Arabic, and Devanagari use different vertical metrics — see the non-Latin scripts note below.

7×9 pixel matrix (ISO 9241-303)

ISO 9241-303 requires that a legible character occupy at least a 7 columns × 9 rows pixel matrix. This is the minimum grid needed to render all distinguishable characters of the Latin alphabet — enough pixels to form ascenders, descenders, counters, and the strokes that separate similar characters (e.g. l, 1, I).

The calculator uses the 9 row (height) constraint as the practical floor: a cap height below 9 dpx is flagged as Below 9 px in the compliance table. The column count is not checked separately — it is generally satisfied by any proportional typeface once the height requirement is met.

At high DPR, 9 dpx can correspond to fewer CSS px (e.g. 3 CSS px at DPR 3), which is why the table shows both values.

Minimum pixel grids for non-Latin scripts

ISO 9241-303 was written with Latin, Cyrillic, and Greek alphabets in mind. Its 7×9 minimum matrix applies to those three scripts. All other major scripts have different structural demands and require higher pixel counts.

The four non-Latin scripts most relevant to digital UIs, by number of users:

CJK — Chinese, Japanese, Korean (~1.3 billion users)

Ideographic characters fill a square em box with far more strokes, intersections, and enclosed spaces than Latin letters. A 7×9 matrix cannot resolve them. Industry practice (GB/T 5199, 1985; modern CJK font tooling) recognises:

  • 12×12 px — threshold of legibility; some characters become unrecognisable at this size
  • 16×16 px — historical minimum; ~99 % of common characters are readable, but some strokes must be simplified
  • 24×24 px — recommended minimum; complex characters and traditional variants survive without stroke omission

At equivalent angular size, a CJK character needs roughly 2–3× more device pixels per dimension than a Latin letter to achieve comparable legibility.

Arabic (~830 million users)

Arabic is a cursive abjad: characters change shape depending on word position (initial, medial, final, isolated), and many share a base form distinguished only by dot diacritics. Research on Arabic dot-matrix displays (IEEE, 1991) established a 5×7 matrix as a floor for symbol identification — not comfortable reading. In practice, dot diacritics become indistinguishable at low resolution even when the base letterform survives, so Arabic requires at least the Latin 7×9 equivalent, and preferably more.

Devanagari (~600 million users — Hindi and ~120 other languages)

Devanagari is an abugida: consonants carry an inherent vowel, and vowel marks are attached above, below, or beside the base character. A horizontal bar (shirorekha) runs across the top of most letters and is used for word segmentation. Consonant clusters form conjunct characters that must remain visually distinct. These stacked forms and diacritics need significantly more vertical pixels than Latin; a minimum of approximately 16 px height is commonly cited for comfortable reading.

Hebrew (~10 million native + 200 million second-language users)

Hebrew is a right-to-left abjad similar in structure to Arabic but with block letterforms rather than cursive. It is less complex than Arabic at low resolutions but shares the diacritic vulnerability (nikud vowel points). The Latin 7×9 matrix is generally considered sufficient for unpointed (niqqud-free) Hebrew text used in most UIs.

Standards gap

No single international standard specifies different arc-minute thresholds or pixel grids for CJK, Arabic, or Devanagari. ISO 24509:2019 covers minimum legible font size across age groups and includes non-Latin provisions in Annex F. DIN 1450 (German legibility standard) addresses stroke-to-height ratios and confusable character pairs but does not specify pixel grids and does not address non-Latin scripts.

This calculator currently applies the ISO 9241-303 Latin/Cyrillic/Greek thresholds only. CJK, Arabic, and Devanagari support may be added in a future version.

Apple iPhone Pro / Pro Max — rendering pipeline exception

Most iOS and iPadOS devices report a devicePixelRatio that matches their hardware exactly — multiplying CSS pixels by DPR gives the true physical resolution. iPhone Pro and Pro Max models are an exception.

These models use a three-layer rendering pipeline. The browser can only see the first two layers — the hardware pixel count is invisible to JavaScript.

Layer iPhone 17 Pro Max example Source
CSS pixels 430 × 932 screen.width × screen.height
Rendered buffer 1290 × 2796 CSS px × DPR (3) — what the GPU draws
Physical panel 1320 × 2868 @ 460 PPI Hardware — not accessible via JS

Because devicePixelRatio is the CSS-to-rendered ratio (3), not the CSS-to-hardware ratio (~3.07), auto-detection reports the rendered resolution rather than the true hardware resolution. This causes a roughly 6 % error in pixel pitch, which affects PPI, arc-minute thresholds, and mm character sizes.

Recommendation: select the matching device from the Preset list, or enter the hardware resolution and diagonal manually.
